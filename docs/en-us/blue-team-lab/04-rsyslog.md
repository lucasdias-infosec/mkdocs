# Implementation Report: Log Centralization with Rsyslog, Auditd, Suricata and Wazuh

## 1. Objective

Implement a centralized architecture for collecting and correlating security events, using Rsyslog as an intermediary layer between log generators (Auditd and Suricata) and the analysis engine (Wazuh).

The objective of this stage was:

- Unify system and network logs.
- Avoid duplicate file reading by multiple agents.
- Separate responsibilities between collection, transport and analysis.
- Prepare the environment for future scalability.

## 2. Architecture Overview

The solution consolidated three security layers into a single server, with clearly defined roles:

- Auditd: monitors events at the kernel level.
- Suricata: performs network traffic inspection (IDS).
- Rsyslog: acts as a log collector and forwarder.
- Wazuh Manager: receives, decodes, correlates and generates alerts in the Dashboard.

Established data flow:

Auditd / Suricata → Rsyslog → Wazuh Manager → Dashboard

## 3. Installation and Service Validation of Rsyslog

The service installation was ensured with:

```bash
sudo apt update
sudo apt install rsyslog -y
```

Validation:

```bash
sudo systemctl status rsyslog
```

The service remained active and enabled for automatic startup.

## 4. Configuration
### 4.1. File Creation

File created:

```bash
/etc/rsyslog.d/99-monitoring.conf
```

### 4.2. File Reading (imfile)

Monitored files:

```bash
/var/log/audit/audit.log
/var/log/suricata/eve.json
```

### 4.3 Forwarding (omfwd)

Forwarding configured to:

```bash
<MANAGER_IP>:515
```

Directive used:

```bash
. @<MANAGER_IP>:515
```

### 4.4. Port Justification

- Port 515/UDP designated for Wazuh.
- Port 514/UDP/TCP reserved for future reception of external logs.

## 5. Wazuh Configuration as Analyzer

In the file:

```bash
/var/ossec/etc/ossec.conf
```

Configured block:

```bash
<remote>
  <connection>syslog</connection>
  <port>515</port>
  <protocol>udp</protocol>
  <allowed-ips><LAB_INTERNAL_NETWORK></allowed-ips>
</remote>
```

## 6. Architectural Adjustment

The <localfile> blocks that performed direct file reading were removed, preventing:

- Log duplication
- Redundant processing
- Internal conflicts

File reading became the exclusive responsibility of Rsyslog.

### 7.2 Network Test (Suricata)

```bash

```

Expected result:

- Signature detection by Suricata
- Record written to eve.json
- Forwarding via Rsyslog
- Alert displayed containing source and destination IP

## 8. Architectural Benefits
### 8.1 Separation of Responsibilities

- Log generation → Specialized tools
- Transport → Rsyslog
- Analysis and correlation → Wazuh

### 8.2. Scalability

New servers can be integrated by simply installing Rsyslog and configuring forwarding to:

```bash
<CENTRAL_SERVER_IP>:514
```

Without requiring changes to the central architecture.

### 9. Architectural Justification

Adopting Rsyslog as an intermediary layer promotes:

- Modularity
- Granular control of log flow
- Preparation for expansion
- Event pipeline organization

The infrastructure evolves toward a structured model of log centralization and correlation, aligned with corporate environment best practices.
